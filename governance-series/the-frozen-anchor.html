The Frozen Anchor

A Document Design Pattern for Long-Lived AI-Assisted Projects

The basic problem is simple: a project can stay coherent sentence by sentence while still becoming a different project over time.

The Frozen Anchor is a design pattern for preventing that drift. It is not a running project summary. It is a prior constraint on what the project is allowed to become. Before repeated LLM-assisted revision begins, the project’s identity is frozen explicitly enough that later narrowing can be judged against it rather than quietly absorbed into it.

This matters because long-lived AI-assisted work does not usually fail through a single dramatic rewrite. It fails through many small accommodations: a narrowed claim here, a dropped variable there, a reframing that looks clarifying in isolation but cumulatively changes the governed object.

A frozen anchor does not stop revision. It distinguishes acceptable narrowing from structural drift.

The pattern

The pattern has four practical components that should live in one authoritative artifact.

Identity statement

A concise statement of what the project is actually about in its irreducible form.

Must-preserve elements

The variables, questions, commitments, or methodological relations that cannot be dropped without changing project identity.

Allowed-to-change list

The forms of narrowing, simplification, or sequencing adjustment that are legitimate.

Drift examples

Concrete examples of the kinds of changes that would look tidy but would actually remake the project.

Why ordinary summaries fail

Most project documents are written as if they should track the latest state of understanding. That makes them good operational summaries but bad identity anchors. They evolve with the project, which means they cannot independently judge whether the project has drifted.

The Frozen Anchor works because it is supposed to become stale in one narrow sense: it should keep stating the original governed identity even while downstream artifacts evolve. That asymmetry is the point.

Acceptable narrowing vs. drift

Acceptable narrowing

A reduction in scope that preserves the governing identity of the project and still answers the same core question under a more bounded frame.

Drift

A change that silently replaces the core question, drops a must-preserve element, or recasts the object so the project is now substantively different.

The distinction matters because long-lived AI-assisted projects often produce narrowing that feels prudent, coherent, and even methodologically stronger. Without a frozen identity anchor, those changes are easy to ratify after the fact.

The three-question test

A practical use test for any major revision is:

What changed?

Name the exact shift rather than calling it general improvement.

Was it licensed?

Check whether the change appears in the allowed-to-change list.

What stayed invariant?

Show how the must-preserve elements are still intact after the revision.

If not, what is it?

If the change cannot clear those tests, treat it as drift or as a proposal for a new project.

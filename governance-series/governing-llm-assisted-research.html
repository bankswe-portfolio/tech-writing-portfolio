A Control Framework for LLM-Assisted Research

Governing Research Definition Under Weak Supervision

When you use an LLM to help you think, who is watching the thinking?

Most writing about LLMs in research workflows focuses on the wrong problem. It asks: how do I get better outputs? Better prompts, better models, better retrieval pipelines. These are real improvements, and they have real value. But they address only the surface of a deeper structural problem — one that better prompts cannot fix, because the problem is not about output quality. It is about governance.

The problem, stated precisely, is this: LLMs are unreliable in a way that is recursively dangerous. They produce confident, coherent, partially correct, hidden-load-bearing falsehoods. They collapse underdetermined human thought into falsely precise premises. They generate elegant structure that feels like rigor without being rigor. And because they do all of this fluently — inside the same channel you use to do everything else — their errors do not announce themselves. They accumulate.

"A single uncaught falsehood can become a premise, then a context assumption, then a design constraint, then a governing rule — and by the time it surfaces, it has contaminated everything downstream."

This is not a description of a bad model. It is a description of what any sufficiently capable language model does by default, because it has no reliable mechanism for self-correction, no external ground truth, and no structural resistance to closed-loop validation. This is the problem that prompting cannot solve. What it requires instead is a control framework.

01 — The failure regime

The Prompting Trap

When a solo researcher uses LLMs heavily for intellectual labor — drafting, reviewing, critiquing, synthesizing — a specific failure regime emerges that has no real analogue in traditional research settings.

In a standard academic environment, progress toward a valid, executable research definition is constrained by external structures: advisor feedback, methodological training, peer challenge, lab norms, institutional review. These structures are noisy and imperfect, but they are external. They provide pressure from outside the researcher's own reasoning. They catch the errors the researcher cannot see.

A solo LLM-assisted researcher has none of that. Instead, they have a tool that is extraordinarily good at producing outputs that feel adequate — polished, plausible, internally coherent — without being genuinely adequate. The tool cannot distinguish between a real improvement and a cosmetic one. It will not flag when a cleaner project has silently substituted for the original one.

The result is a specific failure mode: iterative refinement that produces progressively cleaner documents without producing genuine progress toward a commitment-ready state. Revisions happen. Drafts improve in surface quality. But the same substantive defects persist in relabeled form, the project identity quietly shifts, and there is no mechanism to detect any of it. The solution is not a better prompt. It is external control over the process itself.

02 — The core analogy

A Control-Theory Framing

The most useful analogy for understanding what this framework needs to do comes from engineering. In control theory, a feedback control system has a reference signal — the target state you are trying to achieve. It has a plant — the system being controlled, evolving over time. It has sensors that observe the plant's current state. A controller applies bounded corrective action to reduce the error between current and target state. The whole loop runs continuously, with re-observation after every correction. Critically, a well-designed control system also has anti-windup protections: mechanisms that prevent the controller from applying corrections indefinitely, or mistaking small local changes for genuine convergence. The research governance problem maps onto this structure exactly.

03 — Framework specification

What the Framework Must Do

Translating this structure into working practice requires specifying what the governance system actually needs to accomplish. The following requirements are derived from the failure modes of LLM-assisted solo research — not from preferences about process. They are grouped into four functional pillars.

Pillar I — Foundation

Define and Anchor the Object

  • Name the current artifact set as a bounded, auditable object
  • Freeze a project identity anchor before revision pressure begins
  • Make the authority hierarchy explicit — governing artifacts vs. lineage evidence
Pillar II — Observation

Diagnose Rigorously

  • Separate diagnosis ("what is wrong?") from convergence ("are we progressing?")
  • Require adversarial, staged, bundle-level audit — not prose review
  • Classify defects by severity, class, and locus — not just critique
Pillar III — Control

Act with Precision

  • Maintain a persistent defect register across revision cycles
  • Require every patch to map to a registered defect ID — no unmapped changes
  • Enforce epistemic discipline: separate fact, inference, assumption, speculation
Pillar IV — Convergence

Judge Progress Honestly

  • Gate every cycle for core-identity drift — cleaner ≠ better project
  • Distinguish draft state from committed state explicitly
  • Account for solo-researcher realism and LLM as a control risk

1. Define the governed object explicitly

The system must know what it is governing. That means naming the current artifact set and treating it as an auditable, bounded object — not letting "the project" remain an informal cloud of ideas. Without a defined governed object, there is no stable target for audit, no basis for authority judgments, and no way to detect drift.

2. Freeze a project identity anchor

Before anything else, there must be an immutable statement of project identity: what the work is fundamentally about, what legitimate narrowing looks like, and what counts as drift. This anchor is the reference signal for the entire governance system. Without it, the system can converge on a cleaner project, a narrower project, or an adjacent project — while never detecting that it has ceased to be the original one.

3. Make the authority hierarchy explicit

At any given moment, some artifacts are governing truth and others are lineage evidence. Audit records, historical drafts, prior versions — these are not the same as the current canonical bundle. A governance system must explicitly distinguish them, because a research process that allows earlier conclusions to silently overwrite current commitments has no reliable notion of what it is governing.

4. Separate diagnosis from convergence

These are fundamentally different functions, and conflating them is one of the most common failures in LLM-assisted revision cycles. Diagnosis asks: what is wrong with this artifact right now? Convergence asks: are we actually making progress toward a terminal state, or are we circling? A system that only diagnoses can improve documents indefinitely without ever solving the governance problem.

5. Make audit adversarial, staged, and bundle-level

Unstructured LLM review tends to overproduce descriptive adequacy and underproduce genuine defect detection. This is an empirically documented limitation: LLMs underperform specifically at identifying weaknesses and adjusting feedback to actual quality rather than apparent quality. Useful audit must be deliberately adversarial, staged so phases cannot be collapsed, and must evaluate the research bundle as a coupled system — not isolated prose.

6. Classify defects, not just surface critique

Commentary is not governance. A valid audit must produce structured defect findings: a severity rating, a defect class, a locus, and a downstream significance judgment. Without this structure, audit output cannot drive a defect register, cannot support convergence tracking, and cannot distinguish a project that is improving from one that is merely being criticized in new ways.

7–9. Maintain, patch, and discipline

The governance system needs a persistent defect register — not conversation memory, which is inside the probabilistic channel and cannot be trusted to maintain stable state. It needs bounded patch generation in which every proposed change maps to a registered defect ID. And it needs explicit epistemic discipline throughout, separating what is directly evidenced from what is inferred or assumed.

10. Guard against core-identity drift

The governance system must explicitly ask — at every revision cycle — whether the current bundle still corresponds to the project defined by the frozen anchor. Not whether it is coherent or well-written. Whether it is still the same project. This gate must catch the specific failure mode of LLM-assisted research programs: a bundle that becomes more tractable by quietly abandoning parts of the original project rather than legitimately bounding them.

11–12. Commitment state and solo-researcher realism

Draft state and committed state are not the same. A governance framework that does not explicitly distinguish these states cannot determine whether the bundle is approaching commitment-readiness or merely becoming increasingly polished. For a solo researcher, feasibility is not incidental — the system must ask whether the workload and verification demands are actually within one person's realistic capacity. LLM labor must be treated as a control risk, not a neutral productivity multiplier.

04 — Primary risks

Three Failure Modes

Across all requirements, three failure modes recur as the primary risks in LLM-assisted iterative research definition. Any governance system that does not explicitly guard against all three is incomplete.

Diagram 2 — The Three Failure Modes

Drift


The project quietly becomes a different project. Each revision is locally defensible, but the cumulative trajectory leads away from the original work. Drift is the hardest failure mode to detect because it is distributed across many small revisions, none of which looks wrong in isolation.

Polish Mistaken for Progress

Documents become clearer, more formal, more professionally structured — without becoming more scientifically valid. LLM assistance is particularly good at generating polish, and humans are particularly bad at resisting the impression that a well-formatted document must be a good one.

The Endless Revision Loop

Revisions occur. The defect register is partially cleared. New audits surface new findings. More revisions occur. The system is active, the documents are changing — and no one can answer the governance question: are we converging, or are we circling? Without cross-cycle defect tracking, there is no reliable answer.

Each failure mode is invisible from inside a single revision cycle. They only become detectable when defect state is tracked across cycles — which is why the defect register is structural, not optional.

05 — Stopping conditions

Five Ways to Stop

A governance system without a stopping condition is not a governance system. It is a revision engine. The framework must be able to produce, at each cycle, a single explicit decision state with supporting evidence referencing specific defect register entries — not prose impressions.

Diagram 3 — Governed Decision States
Continue
Revision

Continue Revision

Substantive blocking or major defects remain. The revision path is credible and aligned with the identity anchor. This is not a failure state — it is the normal state of a working research definition process. The system is functioning correctly when it outputs this state and there are real defects to close.

Commit &
Execute

Commit and Execute

Blocking and major issues are exhausted. Cross-artifact coherence is demonstrated. Identity anchor is preserved. Feasibility under realistic conditions is confirmed. The goal state — not perfect documents, but documents that are sufficiently coherent, bounded, and executable to justify binding commitment.

Narrow
Decisively

Narrow Decisively

The current project scope is too broad or insufficiently licensed by the proposed study design. A bounded version still preserves the identity anchor and core research question. This is not failure — it is a legitimate governance output that a solo researcher should be able to act on without loss of project integrity.

Pause for
Design Work

Pause for Unresolved Design Work

Progress cannot be meaningfully assessed because key design commitments remain too underdefined. Further revision before this design work is done would be wasteful at best and misleading at worst. This state prevents the common failure mode of iterating on prose while fundamental methodological questions remain open.

Kill
Current Form

Kill the Current Form

The project, as currently framed, cannot be made commitment-ready without unacceptable distortion or drift from the identity anchor. The hardest state to reach — precisely because LLM-assisted revision can always produce a cleaner version, always find another angle. The framework must be able to say: this revision path is not viable. Continuing it would produce a more polished drift artifact, not a better research program.

Only one decision state is valid per cycle. States are mutually exclusive. The supporting evidence must reference specific defect register entries — not subjective impressions about document quality.

06 — Epistemic honesty

An Honest Note on Limitations

This framework is a governance philosophy, not yet a validated governance instrument.

There is no formal measurement model here. Convergence is defined as cross-cycle defect movement, but the specific observables, thresholds, and stopping rules that would make that assessment reliable and calibrated are not yet specified. There is no auditor calibration protocol — no procedure for checking prompt sensitivity, temporal instability, or evaluator drift across the LLM passes that drive the audit layer.

There is also a bootstrap problem the framework cannot fully resolve from the inside: the same probabilistic system that creates the governance problem is being used, in part, to build the governance solution. A sufficiently external layer — one that tracks ratified premises and defect register state in a deterministic structure outside the LLM's context — would provide stronger guarantees.

What the framework offers, under current conditions, is not a solution to the trust problem. It is a set of structures that make epistemic failure more containable: preventing silent premise hardening, enforcing authority distinctions, constraining unmapped revisions, and making the difference between a chosen narrowing and an unnoticed drift structurally visible rather than merely recoverable in retrospect.

Summary

A solo researcher using LLMs heavily for intellectual labor faces a governance problem, not a prompting problem. LLMs produce errors that are recursively dangerous: confident, coherent, load-bearing, and structurally resistant to self-detection. Standard revision loops amplify rather than correct this risk, because the same tool that generated the problem is used to evaluate the solution.

A proper governance framework treats the problem as a control problem: define the target state, observe current state, compute error, apply bounded corrective action, re-observe, and judge convergence from the movement of the error signal across cycles — not from the surface quality of any individual output.

That framework requires a frozen identity anchor, an explicit authority hierarchy, a staged adversarial audit, a persistent defect register, bounded and traceable patch generation, epistemic discipline throughout, and a convergence layer that can produce one of five explicit decision states: continue, commit, narrow, pause, or kill.

Without that structure, the most likely outcome of sustained LLM-assisted research work is not a bad project. It is an excellent-looking project that is no longer the project you set out to do.

This framework was developed through direct application on a live research program. All requirements were derived from observed failure modes, not from theoretical preferences.

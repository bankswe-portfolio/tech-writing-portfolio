A Governance Architecture for LLM-Assisted Research Programs

A component view of the system behind the governance series

This note is not a fourth essay. It is a companion map of the architecture implied by the other pieces: what the system’s components are, what role each plays, and what remains unbuilt.

The core claim of the series is that long-lived AI-assisted work needs governance infrastructure. This note restates that claim in architectural rather than argumentative form. Instead of focusing on drift, narrowing, or workflow roles separately, it lays out the system as interacting components: identity anchor, authority map, governed object, audit layer, defect memory, bounded revision, re-audit, and convergence judgment.

The point of the architecture is not merely to diagnose defects inside one draft. It is to make cross-cycle movement observable, governable, and interruptible.

The core components

Core Intent Memo

The frozen identity anchor. Defines what the project is, what must remain preserved, and what would count as drift.

Authority Map

Distinguishes current governing artifacts from historical calibration artifacts and support materials.

Research Bundle

The governed object: the canonical proposal and pre-registration under active evaluation.

Audit Instrument

The adversarial diagnostic layer for coherence, feasibility, identity preservation, and inferential discipline.

Defect Register

Persistent memory of defects, each with identity, class, severity, locus, and status across cycles.

Patch Layer

The bounded revision layer that proposes fixes tied explicitly to named defects rather than freeform improvement.

Revision Pass

The actual update to the bundle made through mapped corrective actions rather than uncontrolled rewriting.

Re-Audit Layer

Checks whether a revised bundle actually closes defects or merely restates them in cleaner language.

Convergence Layer

Judges whether the system is progressing, circling, drifting, ready to commit, or ready to stop.

Operator Rules / Runbook

The usage discipline governing freezes, audit order, revision logging, and terminal decisions.

Current build status

Built or substantially implemented

Core Intent Memo, authority distinctions, governed bundle, and a real phased audit instrument already exist in working form.

Specified but incomplete

The defect-register logic, historical calibration discipline, and convergence logic are conceptually defined but not yet fully operationalized.

Still mostly conceptual

The patch layer, re-audit layer, and full operator runbook remain design work rather than completed infrastructure.

Why this architecture matters

Without an explicit architecture, long-lived AI-assisted work defaults to ad hoc iteration. The same context generates, critiques, revises, and implicitly authorizes its own outputs. That is exactly the closed-loop condition the larger series is trying to break.

By making the components explicit, the architecture forces a more honest question: what part of the governance stack actually exists, and what part is still aspiration? That distinction matters if these ideas are going to be taken seriously as working method rather than merely attractive theory.

This note is intentionally conceptual and still evolving. It is included as a companion artifact because it helps situate the rest of the series as a system rather than as four disconnected pieces of writing.
